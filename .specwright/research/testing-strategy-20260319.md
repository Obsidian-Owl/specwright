# Testing Strategy for Specwright

## Context

Specwright is an orchestrator — it doesn't implement domain-specific testing. It guides users through a workflow where the RIGHT tests get written. The problem isn't that tests don't run (they do), but that the tests being written are too often mocked unit tests that don't catch regressions effectively. Integration tests that exercise real boundaries are better at surfacing issues early.

Three questions:
1. How should Specwright guide the tester agent to write better tests (integration over mocked)?
2. How can Specwright detect what testing approach a project needs?
3. Should Specwright ship a meta-skill that helps users create project-specific testing skills/agents?

## Track 1: Domain-Aware Test Strategy Selection

### Detection Heuristics (What Kind of Project Is This?)

Manifest + dependency analysis can identify project domain with high confidence:

| Signal | Domain | Testing Implication |
|--------|--------|-------------------|
| react/vue/angular in deps | Web app | Component integration tests, E2E for critical paths |
| express/fastapi/gin in deps | API | Contract tests, real HTTP request tests (supertest/httpx) |
| `bin` field, clap/cobra deps | CLI | Golden file tests, exit code tests against real CLI |
| .tf files, serverless.yml | Infrastructure | Plan validation, idempotency tests, policy-as-code |
| openapi.yaml, .proto files | API contract | Schema-driven test generation, backward compat checks |
| dbt_project.yml, airflow deps | Data pipeline | Data quality tests, schema validation |
| Docker, k8s manifests | Containerized | Integration tests with real containers (testcontainers) |

**Key insight**: Detection should happen during sw-init (stack detection) and inform the CONSTITUTION.md testing practices. The constitution then guides the tester agent throughout the build.

Sources: go-enry/Linguist detection, Pact docs, Terratest, web.dev testing trophy, k6, dbt-expectations

### Unified Test Type Taxonomy

Six categories that span all domains — useful for spec writing and test quality auditing:

| Category | Tests Answer | Integration Signal |
|----------|-------------|-------------------|
| **Correctness** | Does it produce the right output? | Real inputs, real boundaries |
| **Robustness** | Does it handle errors gracefully? | Real error conditions, not mocked |
| **Performance** | Is it fast enough under load? | Real data volumes, real concurrency |
| **Security** | Is it resistant to abuse? | Real injection attempts, real auth |
| **Compatibility** | Does it work with its deps? | Real dep versions, contract checks |
| **Operational** | Is it observable and deployable? | Real monitoring, real deployment |

Sources: ISTQB taxonomy, web.dev testing strategies, O'Reilly Software Testing Concepts

## Track 2: Writing Better Tests (Integration Over Mocked)

### The Core Problem

The specwright-tester agent currently writes excellent adversarial unit tests with good mutation resistance. But the constitution says "mock only at system boundaries" without guidance on WHEN to prefer integration tests over unit tests. The tester defaults to mocked tests because they're easier to write and faster to run.

### When Integration Tests Beat Mocked Tests

Research converges on this heuristic: **mock when the dependency is expensive/external; integrate when the dependency is your own code.**

| Boundary Type | Test Approach | Why |
|--------------|--------------|-----|
| **Internal module → internal module** | Integration test (no mocks) | Mocks hide contract drift between your own modules |
| **Code → database (owned)** | Integration test (testcontainers or real DB) | Schema changes, query behavior differ from mocks |
| **Code → external API (not owned)** | Mock or contract test | Can't control external API; use recorded responses |
| **Code → filesystem** | Real filesystem (temp dirs) | Filesystem behavior differs across OS; mocks hide this |
| **Code → clock/time** | Inject controllable clock | Time is global state; mocking it is correct |
| **Code → expensive API (cost-attached)** | Mock for unit; real for scheduled integration | Cost makes real calls impractical per-commit |

**The Testing Trophy model** (web.dev): Integration tests should be the LARGEST layer, not unit tests. The classic pyramid is outdated for modern systems.

Sources: Pact docs, web.dev testing strategies, Testcontainers, SWE-bench PASS_TO_PASS methodology

### What Should Change in Specwright

**In the constitution (generated by sw-init):**
- Add domain-specific testing guidance based on detected stack
- E.g., for an Express API: "Prefer supertest integration tests over mocked request/response objects. Use real database for data access tests."

**In the tester agent's behavior:**
- When writing tests for code that calls other internal modules, write integration tests (import real modules, not mocks)
- When writing tests for code that talks to a database, suggest testcontainers or real test DB
- Reserve mocks for: external APIs you don't control, expensive/rate-limited services, time/randomness

**In the spec review:**
- Flag acceptance criteria that can only be verified via integration test but are likely to get mocked
- E.g., "data persists across requests" — this MUST be an integration test, not a mock

### The Regression Design Angle

The issue isn't that regression tests don't run — it's that mocked unit tests don't CATCH regressions effectively. A mocked test can pass even when the real integration is broken.

**Integration tests are better regression detectors because:**
1. They exercise real module boundaries (where regressions happen)
2. They catch contract drift between modules (mocks freeze the contract at mock-time)
3. They catch behavioral changes in dependencies (mocks don't change when deps change)

**Property-based testing** (Hypothesis/QuickCheck) is particularly effective for regression detection because it tests invariants rather than specific outputs — making tests resilient to incidental changes while catching real behavioral regressions. An arXiv paper (2510.09907) demonstrates LLM agents can infer and generate property-based tests automatically.

Sources: SWE-bench FAIL_TO_PASS + PASS_TO_PASS, Pact consumer-driven contracts, aider --auto-test, Agentic Property-Based Testing (arXiv 2510.09907)

## Track 3: The Meta-Skill Approach

### Specwright as Orchestrator, Not Implementer

Specwright doesn't need to ship domain-specific testing agents. It needs to:
1. **Detect** what the project needs (sw-init already does stack detection)
2. **Guide** the tester to write the right kind of tests (constitution + tester agent prompt)
3. **Enable** users to create or install testing skills/agents specific to their domain

### A "Testing Strategy" Skill

A new skill or enhancement to sw-init that:
1. Analyzes the project (deps, frameworks, architecture patterns)
2. Generates a `testing-strategy` section in the constitution with domain-specific guidance
3. Recommends creating project-local testing agents (in `.claude/agents/`) for domain-specific concerns
4. Optionally generates starter testing agents based on detected patterns

Example: for a detected Express + PostgreSQL API:
```markdown
## Testing (generated by stack detection)
- HTTP endpoint tests use supertest against a real Express app (not mocked req/res)
- Database tests use a test database (not mocked queries)
- External API calls (Stripe, SendGrid) are mocked with recorded responses
- Contract tests verify API responses match the OpenAPI spec
```

### Agent Specialization: Hub-and-Spoke, Not Monolith

Research strongly supports splitting testing concerns across focused agents (Anthropic, Azure, OpenAI all converge). But Specwright doesn't need to ship all the specialists — it ships the orchestration pattern and lets users add domain specialists:

**What Specwright ships (the orchestrator):**
- specwright-tester (writes adversarial tests — enhanced with integration-vs-mock guidance)
- Gate pipeline (build → tests → security → wiring → spec)
- The ability for users to add project-local agents

**What users create or install (domain specialists):**
- `.claude/agents/api-tester.md` — knows about contract testing, load testing
- `.claude/agents/infra-tester.md` — knows about policy-as-code, idempotency
- `.claude/agents/security-tester.md` — knows about OWASP, injection patterns
- Or: installed from a plugin marketplace as testing-focused plugins

**The key insight from skill-creator**: Execution and evaluation are distinct cognitive tasks. The tester writes tests (execution). The gate-tests audits test quality (evaluation). These are already separate in Specwright — the architecture is right. The gap is in the GUIDANCE the tester receives about what kind of tests to write.

Sources: Anthropic building-effective-agents, Anthropic multi-agent-research-system, Anthropic context-engineering, OpenObserve QA case study, Azure agent orchestration patterns, skill-creator SKILL.md

## Synthesis: What Should Change

### Priority 1: Enhance the tester agent's integration test guidance
- Update specwright-tester.md to prefer integration tests for internal boundaries
- Add the "mock when external, integrate when internal" heuristic to the tester's prompt
- Constitution generation (sw-init) should include domain-specific testing practices

### Priority 2: Domain-aware constitution generation
- sw-init already detects the stack — extend it to generate testing-specific constitution rules
- E.g., detected Express → "use supertest"; detected Terraform → "test idempotency"
- This is a refinement of sw-init, not a new skill

### Priority 3: Enable project-local testing agents
- Document how users can create `.claude/agents/` specialists for their domain
- Consider a meta-skill (or sw-guard enhancement) that recommends testing agents based on detected gaps
- Consider shipping starter agent templates for common domains (API testing, data pipeline testing, infra testing)

### What NOT to do
- Don't ship 8 domain-specific tester agents in core/ — users build or install what they need
- Don't add new gates for every test type — the existing gate pipeline is sufficient
- Don't over-complicate the regression story — tests run fine, they just need to be better designed

## Caveats

1. Domain detection is probabilistic — multiple domains coexist. Surface ambiguity to user.
2. Integration tests are slower than mocked tests — the constitution should set expectations.
3. Property-based testing via LLM agents (arXiv 2510.09907) is promising but post-cutoff.
4. Embedded/mobile testing strategies have limited research coverage.
5. The meta-skill approach is untested — it's a design hypothesis, not a validated pattern.
