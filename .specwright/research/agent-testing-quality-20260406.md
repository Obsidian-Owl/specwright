# Research Brief: Agent Harnesses and Quality Testing in SWE Agentic Coding (2026)

Topic-ID: agent-testing-quality
Created: 2026-04-06
Updated: 2026-04-06
Tracks: 5

## Summary

Deep external analysis of how SWE agent frameworks handle test quality, tier selection, live infrastructure, and multi-step deliverable validation. Key takeaways: (1) No agent framework has native test tier selection — all default to mock-heavy unit tests, confirmed by a 1.2M-commit empirical study. (2) Testcontainers has published official Claude Skills, and Specmatic/PactFlow have released MCP servers for contract testing — these are the most mature agent-compatible live testing tools. (3) No multi-agent framework has a "deliverable-level verification" step — all verify per-step, not per-outcome. (4) The primitives exist (subagents with Bash, agent teams, hooks, MCP servers) but no tool assembles them into an integrated testing strategy. (5) No published system specializes agents by test pyramid level — this would be novel. Claude Code subagents natively support per-agent skill injection, making it technically feasible. (6) No test pyramid health scanner or testing strategy drift detector exists — both would be first-of-kind.

## Findings

### Track 1: Agent Harness Testing Architectures

#### F1: No SWE agent has native test tier selection
- **Claim**: Across Claude Code, Cursor, Windsurf, Cline, Aider, OpenCode, Codex CLI, and Devin, no tool has a built-in mechanism that analyzes code context and selects integration over unit tests.
- **Evidence**: Comprehensive survey of official documentation for all 8 tools. The closest is a community-authored `windsurf-test-generation` skill that allows configuration to "prefer unit vs integration tests" — but this is not Windsurf core. Claude Code hooks can enforce test presence but don't select tiers.
- **Source**: https://code.claude.com/docs/en/skills, https://code.claude.com/docs/en/agent-teams, https://docs.windsurf.com/windsurf/cascade/cascade
- **Confidence**: HIGH
- **Potential assumption**: No

#### F2: Agents generate 36% more mock-heavy test commits than humans
- **Claim**: A January 2026 empirical study of 1.2M+ commits across 2,168 repositories found that agents added mocks to 36% of test commits vs. 26% for non-agents. Agents relied on a single mock type (`mock`) 95% of the time, while non-agents used greater variety: mock (91%), fake (57%), spy (51%).
- **Evidence**: "Coding agents predominantly use the mock type (95%), whereas non-agents employ a wider variety: mock (91%), fake (57%), and spy (51%)"
- **Source**: https://arxiv.org/html/2602.00409v1
- **Confidence**: HIGH
- **Version/Date**: January 2026
- **Potential assumption**: No

#### F3: Claude Code agent teams support cross-layer test coordination
- **Claim**: Agent teams (experimental, v2.1.32+) allow multiple Claude Code instances to coordinate via shared task list and direct messaging. The official use case list includes "cross-layer coordination: changes that span frontend, backend, and tests, each owned by a different teammate."
- **Evidence**: Direct from official documentation. A teammate can own the test layer. `TaskCompleted` hooks with exit code 2 block task completion until quality criteria are met.
- **Source**: https://code.claude.com/docs/en/agent-teams
- **Confidence**: HIGH
- **Version/Date**: v2.1.32+, experimental
- **Potential assumption**: No

#### F4: SWE-CI benchmark measures long-horizon regression introduction
- **Claim**: SWE-CI (March 2026) evaluates agent capabilities in CI maintenance over time (100 tasks, avg 233 days history, 71 commits/task). Most models achieve zero-regression rates below 0.25 — agents frequently introduce regressions during long-horizon maintenance.
- **Evidence**: Published benchmark paper. First benchmark that explicitly measures test-maintenance quality, not just patch correctness.
- **Source**: https://arxiv.org/abs/2603.03823
- **Confidence**: HIGH
- **Version/Date**: March 2026
- **Potential assumption**: No

#### F5: Meta's Just-in-Time Testing generates tests per-change, not per-suite
- **Claim**: Meta Engineering published JiTTests (February 2026) — tests generated by LLMs on-the-fly for specific code changes rather than maintained as a static suite. No test maintenance required, no test code review, tailored to the change.
- **Evidence**: "Just-in-Time Tests are automatically generated by large language models on the fly to catch bugs... No test maintenance required."
- **Source**: https://engineering.fb.com/2026/02/11/developer-tools/the-death-of-traditional-testing-agentic-development-jit-testing-revival/
- **Confidence**: HIGH
- **Version/Date**: February 2026
- **Potential assumption**: Yes — Meta's approach is internal/research, not available as a tool

#### F6: SWE-bench rankings are unreliable as quality proxies
- **Claim**: UTBoost research found 26/500 SWE-bench Verified tasks had insufficient tests. 24% of leaderboard rankings changed when tests were corrected. Roughly one-third of issues contain solutions in the issue text.
- **Evidence**: Published analysis showing benchmark instability when test quality is improved.
- **Source**: https://www.verdent.ai/blog/swe-bench-verified-technical-report, https://medium.com/@danieldkang/swe-bench-verified-is-flawed-despite-expert-review-utboost-exposes-gaps-in-test-coverage-4b75c6b940c6
- **Confidence**: HIGH
- **Potential assumption**: No

#### F7: Open-source AI agent frameworks test infrastructure deterministically, skip planning logic
- **Claim**: Empirical study of agent frameworks found mock assertions in ~41% of test functions. Deterministic components (tools, parsers) get 29.7-40.1% coverage; FM-based planning logic gets less than 5%. Novel FM-testing methods (DeepEval) show very low adoption.
- **Evidence**: Ten testing patterns identified, all conventional unit-test patterns adapted for non-determinism.
- **Source**: https://arxiv.org/html/2509.19185v1
- **Confidence**: HIGH
- **Potential assumption**: No

### Track 2: Subagent and Skill Patterns for Test Infrastructure

#### F8: Testcontainers has published official Claude Code skills
- **Claim**: The Testcontainers organization maintains `github.com/testcontainers/claude-skills` with two published skills: `testcontainers-go` (62+ modules) and `testcontainers-dotnet` (65+ modules). Skills teach Claude to provision real Docker-backed services (PostgreSQL, Redis, Kafka, MySQL, MongoDB) as part of test setup.
- **Evidence**: Direct repository inspection. Skills exist specifically to prevent agents from defaulting to mocks.
- **Source**: https://github.com/testcontainers/claude-skills
- **Confidence**: HIGH
- **Potential assumption**: No

#### F9: Subagents can provision real infrastructure via Bash but cannot nest
- **Claim**: Subagents with `Bash` in their tools array can run `docker run`, `docker compose up`, `kubectl apply`, etc. However, subagents cannot spawn their own subagents — "Don't include `Agent` in a subagent's tools array." The parent must orchestrate infrastructure provisioning and test execution as separate steps.
- **Evidence**: "Subagents cannot spawn their own subagents."
- **Source**: https://platform.claude.com/docs/en/agent-sdk/subagents
- **Confidence**: HIGH
- **Potential assumption**: No

#### F10: Claude Code skills support `context: fork` for isolated infrastructure
- **Claim**: The `context: fork` frontmatter field runs a skill in an isolated subagent. Combined with `allowed-tools: Bash`, a skill can provision containers, run tests, clean up, and return results without polluting main context. Dynamic shell injection via `` !`command` `` syntax executes as preprocessing.
- **Evidence**: Official skills documentation describes the fork context and shell injection patterns.
- **Source**: https://code.claude.com/docs/en/skills
- **Confidence**: HIGH
- **Potential assumption**: No

#### F11: No existing skill/pattern encodes the "real vs mock" decision
- **Claim**: No canonical skill, library, or pattern was found that causes an agent to read project testing configuration (e.g., TESTING.md) and switch between real infrastructure and mocks based on context. The gap identified in the Specwright retro is confirmed as an industry-wide absence.
- **Evidence**: Searched Claude Code skills ecosystem, official docs, community sources. AWS architecture blog describes a hybrid model as practitioner guidance, not agent-executable logic.
- **Source**: https://aws.amazon.com/blogs/architecture/architecting-for-agentic-ai-development-on-aws/
- **Confidence**: HIGH
- **Potential assumption**: No

#### F12: Docker-in-Docker is a last resort; sibling containers are preferred
- **Claim**: Testcontainers documentation explicitly characterizes DinD as "generally considered an instrument of last resort." The recommended alternative is Docker Wormhole/sibling containers: mount host Docker socket and run containers as siblings.
- **Evidence**: "Docker-in-Docker is generally considered an instrument of last resort."
- **Source**: https://java.testcontainers.org/supported_docker_environment/continuous_integration/dind_patterns/
- **Confidence**: HIGH
- **Potential assumption**: No

### Track 3: Quality Assurance in Multi-Step Agent Workflows

#### F13: No multi-agent framework has deliverable-level verification
- **Claim**: Across CrewAI, AutoGen, LangGraph, and Claude Agent SDK, no framework provides a built-in mechanism for validating that independently-produced components work together as a cohesive whole. All verify per-step. CrewAI guardrails are task-scoped. LangGraph provides interrupt-based phase checks but leaves boundary selection to implementers. AutoGen's AgentEval is a three-agent critic system (CriticAgent, QuantifierAgent, VerifierAgent) but not wired into build workflows by default.
- **Evidence**: Comprehensive survey. "The responsibility for defining and running integration tests across combined WU outputs falls entirely to the implementer" — consistent across all frameworks.
- **Source**: https://www.analyticsvidhya.com/blog/2025/11/introduction-to-task-guardrails-in-crewai/, https://docs.langchain.com/oss/python/langchain/human-in-the-loop, https://microsoft.github.io/autogen/0.2/blog/2024/06/21/AgentEval/
- **Confidence**: HIGH
- **Potential assumption**: No

#### F13a: Three-agent QA system achieves 60% reduction in invalid tests — but specializes by pipeline stage, not test type
- **Claim**: "The Rise of Agentic Testing" (arXiv 2601.02454, January 2026) introduces a closed-loop system of Test Generation, Execution and Analysis, and Review and Optimization agents. Demonstrates 60% reduction in invalid tests and 30% coverage improvement over single-model baselines. IMPORTANT: The specialization is by pipeline role (generate → execute → review), NOT by test category (unit vs. integration vs. contract). No published system uses separate agents per test pyramid level.
- **Evidence**: "A closed-loop, self-correcting system in which a Test Generation Agent, an Execution and Analysis Agent, and a Review and Optimization Agent collaboratively generate, execute, analyze, and refine tests until convergence."
- **Source**: https://arxiv.org/abs/2601.02454
- **Confidence**: HIGH
- **Version/Date**: January 2026
- **Potential assumption**: No

#### F13b: Self-filtering via tests doubles agent precision to 47.8%
- **Claim**: SWT-Bench (NeurIPS 2024) found that when SWE-Agent filtered code repairs to only those passing self-generated tests, precision doubled to 47.8%. Test generation and fix generation are weakly correlated tasks, but test-based filtering is a strong quality signal.
- **Evidence**: "Reproducing an issue with a test and fixing this issue are distinct tasks of different difficulty."
- **Source**: https://arxiv.org/html/2406.12952v3
- **Confidence**: HIGH
- **Version/Date**: NeurIPS 2024
- **Potential assumption**: No

#### F13c: SWE-CI shows agents achieve <0.25 zero-regression rate in long-horizon work
- **Claim**: SWE-CI (March 2026) is the first CI-loop benchmark. Most models achieve zero-regression rates below 0.25 — agents routinely break previously passing tests during extended development. Introduces EvoScore to measure maintainability of early design decisions across later modifications.
- **Evidence**: "Although LLMs have shown significant improvements in snapshot-based code modification tasks, they still face substantial challenges in fully automated, long-term, and multi-round software development and maintenance scenarios."
- **Source**: https://arxiv.org/html/2603.03823v1
- **Confidence**: HIGH
- **Version/Date**: March 2026
- **Potential assumption**: No

#### F13d: BriefingScript proposes encoding validation strategy in planning briefs
- **Claim**: The Agentic SE Roadmap paper (arXiv 2509.06216) introduces BriefingScript with a "Validation Loop" that codifies "a multi-level validation strategy from linting and unit tests to integration tests" directly into planning briefs. Also proposes "Merge-Readiness Packs" requiring proof of functional completeness in realistic scenarios.
- **Evidence**: "Encodes the acceptance testing plan directly into the brief with a multi-level validation strategy."
- **Source**: https://arxiv.org/html/2509.06216v1
- **Confidence**: HIGH
- **Version/Date**: September 2025
- **Potential assumption**: No

#### F13e: The self-consistency trap defeats self-verification without external tools
- **Claim**: Models exhibit a "self-consistency trap" — they "confidently defend wrong answers through multiple reflection rounds without external validation." Tool-grounded verification (running tests, not just self-critique) is the prescribed mitigation.
- **Evidence**: "Ground reflection in external tools and data sources rather than relying purely on the model's internal judgment." — Zylos Research, March 2026
- **Source**: https://zylos.ai/research/2026-03-06-ai-agent-reflection-self-evaluation-patterns
- **Confidence**: MEDIUM (secondary synthesis, but underlying papers are peer-reviewed)
- **Potential assumption**: No

#### F13f: Tweag Handbook explicitly calls for integration tests across work unit boundaries
- **Claim**: "When major function groups are built for files and functions that need to work together, an integration test should be built to prove that the entire process and data flow works properly between the files and functions." This is the clearest industry statement about planning integration tests across work units.
- **Evidence**: Direct quote from the Agentic Coding Handbook implementation plan prompt.
- **Source**: https://tweag.github.io/agentic-coding-handbook/examples-prompts/create-implementation-plan/PROMPT/
- **Confidence**: HIGH
- **Potential assumption**: No

#### F13g: Gartner projects 40%+ agentic AI project cancellation by 2027
- **Claim**: Gartner forecasts that over 40% of agentic AI projects will be canceled by end of 2027 due to "escalating costs, unclear business value, and inadequate risk controls" — with inadequate validation as a primary cited risk.
- **Evidence**: Industry analyst projection.
- **Source**: https://virtualassistantva.com/news/ai-workflow-orchestration-multi-agent-systems-1445-percent-inquiry-surge-enterprise-2026
- **Confidence**: MEDIUM (secondary source reporting Gartner)
- **Potential assumption**: No

### Track 4: Live Testing Infrastructure Automation

#### F14: Specmatic MCP Server enables a self-correcting contract test loop
- **Claim**: Specmatic released an MCP server exposing contract testing and mock server capabilities to coding agents. The workflow: agent generates code → Specmatic runs contract tests against real service → failures returned as structured feedback → agent revises → repeat until pass → generate frontend against spec-compliant mock → validate with Playwright.
- **Evidence**: "The LLM reads these failures, understands the points of contention — such as the validation of data types — and attempts to rectify them."
- **Source**: https://specmatic.io/updates/build-apps-from-api-specs-using-ai-self-correcting-contract-driven-agentic-workflows-with-specmatic/
- **Confidence**: HIGH
- **Version/Date**: 2025-2026

#### F15: PactFlow MCP Server brings consumer-driven contracts to agents
- **Claim**: PactFlow released an MCP server for IDE-native contract testing. Agents can generate Pact tests from OpenAPI specs, HTTP traffic, or source code; update tests for code changes; review and fix using AI-trained Pact practices. Claims "up to 60% faster test creation and maintenance."
- **Evidence**: Official PactFlow blog announcement.
- **Source**: https://pactflow.io/blog/pactflow-mcp-server/
- **Confidence**: HIGH

#### F16: Encore's "always-real" model eliminates the mock/real decision

### Track 5: Specialized Test Subagents and Test Infrastructure Audit

#### F21: No published system uses separate agents for unit vs. integration vs. contract tests
- **Claim**: Across academic literature, commercial QA vendors, Kiro, and Specmatic, zero examples found of a deployed system routing test generation to different agents based on test pyramid level. Multi-agent QA systems specialize by pipeline phase (generate/execute/review) or by application layer (UI/API/DB/security), never by test type.
- **Evidence**: Commercial systems like Virtuoso QA use 5 agents (UI Testing, API Testing, Database Testing, Security Testing, Performance Testing) — layer specialization, not tier specialization. Kiro generates unit, integration, and API tests from specs but uses a single agent flow.
- **Source**: https://www.virtuosoqa.com/post/multi-agent-testing-systems-cooperative-ai-validate-complex-applications, https://kiro.dev/docs/specs/
- **Confidence**: MEDIUM (absence is hard to prove definitively)
- **Potential assumption**: Yes — proprietary systems may exist undisclosed

#### F22: Claude Code subagents support per-agent skill injection, enabling differentiated test agents
- **Claim**: Subagent definitions accept a `skills` frontmatter field that injects full skill content into the subagent's context at startup. Subagents don't inherit parent skills. Combined with per-agent `tools`, `mcpServers`, and `permissionMode`, each test agent can have entirely different capabilities.
- **Evidence**: "skills: Skills to load into the subagent's context at startup. The full skill content is injected, not just made available for invocation. Subagents don't inherit skills from the parent conversation." Plugin subagents do NOT support `hooks`, `mcpServers`, or `permissionMode` — only project and user subagents do.
- **Source**: https://code.claude.com/docs/en/sub-agents
- **Confidence**: HIGH
- **Potential assumption**: No

#### F23: No automated test pyramid health scanner tool exists
- **Claim**: No dedicated tool scans a codebase and reports on test type ratios, mock density, or infrastructure usage. SonarQube and Parasoft measure line coverage only. ## Conflicts & Agreements

**Agreement across all tracks**: The mock-default problem is industry-wide, not Specwright-specific. F2 (empirical study), F7 (framework testing practices), F11 (no real-vs-mock decision pattern), and F13 (no deliverable verification) all confirm the same structural gap. F13c (SWE-CI) adds temporal evidence: agents don't just mock too much, they also break existing tests at a 75%+ rate during long-horizon work.

**Agreement**: Contract testing MCP servers (F14, F15) and Testcontainers skills (F8) are converging on the same solution: give agents access to real infrastructure and structured failure feedback. Specmatic's self-correcting loop and PactFlow's IDE-native contracts approach this from different angles.

**Agreement**: F13b (SWT-Bench) and F13e (self-consistency trap) converge: self-critique is unreliable, but test-based filtering is powerful. Running real tests doubles precision. This strongly supports Specwright's gate model — external tool-grounded verification, not just LLM self-review.

**Agreement**: F13d (BriefingScript Validation Loop) and F13f (Tweag Handbook) both advocate encoding integration test requirements in the planning phase, before implementation begins. This directly validates the retro's recommendation to tag ACs with minimum testing tiers during sw-plan.

**Tension**: Meta's JiTTests (F5) vs. the test pyramid model. JiTTests argues for per-change disposable tests with no maintenance burden. This conflicts with the pyramid model where E2E/integration tests are persistent regression guards. Both may be valid for different contexts — JiTTests for rapid change detection, pyramid for system confidence.

**Tension**: Agent team hooks (F3) can enforce test quality gates, but agent teams are experimental with known limitations (task status lag, no nested teams, skills not applied to teammates). The enforcement mechanism exists but may not be reliable enough for production use.

**Agreement**: Encore's always-real model (F16) and Testcontainers' programmatic API (F8) represent the same philosophy: make real infrastructure cheap enough that mocking provides no benefit. The difference is scope — Encore is framework-level, Testcontainers is library-level.

## Open Questions

1. **Can the Testcontainers Claude Skills be extended to Python and Java?** Only Go and .NET are published. Python (for Floe) and Java are gaps.

2. **How stable are agent teams for production use?** Known limitations (task status lag, no nested teams) may block using teams for coordinated test infrastructure + test execution workflows.

3. **Does the Specmatic MCP self-correcting loop work with Claude Code specifically?** Documentation mentions Claude Code but no detailed integration guide or success metrics were found.

4. **What is the actual cost impact of running real infrastructure in agent test loops?** No study quantifies the token + compute + infrastructure cost of integration testing vs. mock-only testing in agent workflows.

5. **How do agent-generated contract tests compare to human-generated ones?** PactFlow claims 60% faster creation, but no quality comparison exists.

6. **What happened to the SWE-agent infrastructure approach?** SWE-agent is maintenance-only; SWE-World proposes LLM-simulated environments. Is the simulation approach viable for testing, or does it reintroduce the mock problem at a different layer?

7. **How does EvoScore (SWE-CI) relate to Specwright's gate model?** EvoScore rewards early design decisions that facilitate later modifications. Could Specwright use a similar metric to evaluate plan decomposition quality — i.e., did the WU decomposition make integration easy or hard?

8. **What is the practical implementation path for BriefingScript's Validation Loop?** The concept (encode multi-level validation strategy in the plan) maps to sw-plan, but the paper is theoretical. Has anyone implemented this in a production agent workflow?

9. **How effective is the three-agent QA system (arXiv 2601.02454) on real-world codebases?** The 60% invalid test reduction was demonstrated on microservice applications. Does it generalize to the project types in Specwright's user base?
